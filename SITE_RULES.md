## 0) 本檔案是「網站法律」（必遵守）

### 0.1 用語定義

* 我說的「法律 / 網站規則 / 規定」= **執行任何修改時一定要遵守的規矩**
* 本檔案（SITE_RULES.md）是全站唯一的規則來源之一，優先級最高
* RELEASE_CHECKLIST.md 是每次改版/發布前的驗收流程（必做）

### 0.2 修改本專案的標準執行程序（任何人都必須照做）

1. 修改前：完整閱讀

   * `SITE_RULES.md`
   * `RELEASE_CHECKLIST.md`
2. 修改中：所有改動必須符合本法律
3. 修改後：必須完成驗收（見 RELEASE_CHECKLIST 與本檔各節的「驗收條件」）
4. 未遵守：視為無效修改

   * 必須撤回（revert）或重新修改直到符合規則

### 0.3 本法律適用範圍

* 包含但不限於：新增頁面、修改 JS/CSS、改資料夾/檔名、調整索引規則、更新 assets、修改類別系統等

### 0.4（補充）規則優先級

* 若本法律與其他任何說明文件、註解、口頭約定衝突：**以本法律為準**
* 若本法律內不同條文互相衝突：以「較高節號的專門規則」優先（例如 C++ 特規優先於全站總則），並應立即修正文句避免再次衝突

### 0.5 驗收條件（必做）

1. 修改前確實閱讀 `SITE_RULES.md` 與 `RELEASE_CHECKLIST.md`
2. 修改後能指出本次改動對應的法律條文（可引用條號）
3. 若未符合任何一條「必遵守」：必須撤回或修到符合為止

### 0.6（新增法律）模板類修改的跨檔案一致性驗收（必做）

* 只要本次修改涉及以下任一項：

  * header / banner
  * breadcrumb
  * 類別按鈕（`.nav-action-link`）
  * 題目連結格式
  * 文章日期規則
  * metadata / 類型 / 難度顯示
  * TOC 模板
  * 文章資訊卡模板
* 就不得只修改單一頁面後直接視為完成。
* 必須至少做：

  1. **同類型頁面全掃描**
  2. **同資料夾內頁面比對**
  3. **舊模板殘留檢查**
  4. **對應 all problems / contests 雙版本檢查（若適用）**
* 若仍存在舊格式殘留，則本次模板修改視為未完成。

### 0.7（新增法律）結構性錯誤修正後，必須同步升級法律或驗收流程

* 若本次修正的是「可重複發生的結構性錯誤」，不得只修檔案本身。
* 必須同步更新：

  * `SITE_RULES.md`
  * 或 `RELEASE_CHECKLIST.md`
* 這類結構性錯誤包含但不限於：

  * 題單首頁漏列文章
  * `site-index.json` 未同步更新
  * `folderCounts` 數字與實際文章不一致
  * 題單頁排序錯誤
  * 複製文章後 breadcrumb / permalink / 註解殘留舊分類
  * 題單文章頁第一行註解不一致
* 若只修結果、不修流程，視為修正不完整。

## 1) 類別系統（Categories / Index）總原則（全站規則）

### 1.1 類別索引來源

* 類別索引是從「全站所有文章頁面」內的 `.post-tag` 元素建立
* 只要是文章頁面，且有 `.post-tag`，就會被納入類別索引

### 1.2 全站索引範圍（重要）

* 預設索引範圍：**全站所有文章**
* 包含但不限於：

  * `/cpp-notes/`
  * `/home/`
  * `/notes/`
  * 其他獨立文章頁（例如經驗分享、學習心得）

### 1.3 非 C++ 文章

* 非 C++ 的文章（例如 `home/ntu-math-special.html`）
* 只要有 `.post-tag`，就應該正常出現在類別系統中
* 這些文章不受 C++ 學習筆記的過濾規則影響

### 1.4 文章被索引的必要條件（類別標準）

* 文章頁面必須包含 `.post-tag` 類別標籤（文章資訊區塊顯示用的那些 tags）。
* 沒有 `.post-tag` 的頁面不視為可索引文章。

### 1.5 驗收條件（必做）

1. 任一應被收錄的文章頁都能在 `site-index.json` 找到
2. 類別頁（categories.html / category.html）顯示的類別與文章數量與 `site-index.json` 一致
3. 任一文章頁若缺 `.post-tag`，必須被視為「未收錄」（且需修正文章頁）

## 2) C++ 學習筆記的特殊規則（關鍵）

### 2.1 類別頁面的顯示範圍

* **只有在 C++ 學習筆記相關頁面中**（例如 category.html）
* 類別文章清單才有額外限制

### 2.2 C++ 類別頁的預設規則

* 預設只顯示來源為：

  * `/cpp-notes/all problems/` 內的文章
* 其他 C++ 子資料夾（例如 grind75 / leetcode75 / neetcode150 / contests...）：

  * 預設不會顯示在 C++ 類別文章清單中
  * 除非我明確指定要納入

### 2.3（補充）C++ 特規只影響「C++ 類別頁的顯示/計數」，不影響全站索引

* 全站類別索引仍以「全站文章頁」為準（見第 1 節）
* C++ 特規只是在 C++ 筆記場景（scope=cpp-notes）下做過濾顯示（見第 6 節）

### 2.4 驗收條件（必做）

1. 全站類別系統仍能看到非 C++ 文章（如 home/xxx.html）
2. C++ 類別頁（scope=cpp-notes）只顯示 `/cpp-notes/all problems/` 的文章（除非你明確指定納入其他資料夾）
3. contests / grind75 等資料夾文章即使不顯示在 C++ 類別頁，也不得被從全站索引中移除

## 3) 新增文章的基本規範（重要）

### 3.1 新增文章的語意定義

* 當我說：
  -「新增文章」
  -「新增文章頁面」
  -「新增一篇題解」
  -「新增某題的文章」
* 預設意思為：
  **依照既有文章（Two Sum）的格式，新建一篇完整文章頁面**

若只需要修改既有文章，我會明確說「修改文章」或「更新內容」。

### 3.2 每一篇新文章必須包含（以 Two Sum 為標準）

* 文章本體內容（解題說明 / 經驗說明）
* 文章資訊區塊（例如：題號、標題、分類、標籤）
* 文章目錄（TOC，若文章長度足夠）
* `.post-tag`（供類別系統索引使用）

> 新增的文章頁面**一定要**包含上述結構（尤其是 `.post-tag` 與文章資訊區塊），並且**會被納入**類別系統索引與統計。

### 3.3（補充，依你最新回答）文章頁定義：內文 + TOC + 文章資訊方塊缺一不可

* 只要我說「文章頁面」，預設表示：

  * **一定要有：文章內文（文章內容區）**
  * **一定要有：目錄 TOC**
  * **一定要有：文章資訊方塊（文章資訊卡）**
* 缺一不可，除非我明確要求「其他格式」

### 3.4（新增法律）單篇文章頁上方 header / banner 模板必須統一

* 所有**單篇文章頁**的上方 header / banner 模板必須一致。
* 這裡的「一致」包含但不限於：

  * site header 外觀與結構
  * breadcrumb 容器
  * hero 區塊主要結構
  * 右上角「類別」按鈕 `.nav-action-link`
  * icon 結構與文字格式
* **只允許的差異**：

  1. 路徑層級差異（例如 `../../` 與 `../../../`）
  2. breadcrumb 因資料夾不同而產生的路徑文字差異
* 不允許的差異（明確禁止）：

  * 有的頁面只有純文字 `類別`
  * 有的頁面用另一套 SVG icon
  * 有的頁面用 `<span>類別</span>`、有的直接寫 `類別`
  * 有的頁面 header 結構與其他文章頁不同
* 規則本質：
  **單篇文章頁模板要以「主流正確模板」為唯一標準，不得混用多套版本。**

### 3.5（新增法律）模板一致性不只看畫面，也看 HTML 結構

* 所謂「同模板」，不只是肉眼看起來差不多。
* 若屬於同一模板的頁面，其 HTML 結構應盡量一致，尤其是：

  * `.nav-action-link`
  * breadcrumb
  * hero 區塊
  * article meta 容器
* 禁止只做到「視覺近似」但底層 HTML 明顯不同。

### 3.6 驗收條件（必做）

1. 新增的頁面符合「文章頁定義」（內文 + TOC + 文章資訊方塊）
2. 文章頁面包含 `.post-tag`，且可被 `site-index.json` 收錄
3. 單篇文章頁的上方 header / banner 模板符合全站主流模板
4. 不符合者不得稱為「新增文章」，必須補齊結構

## 4) C++ 筆記文章放置規範（關鍵）

### 4.1 all problems 為「必定收錄」資料夾

* **所有 C++ 解題類文章**

  * 預設「一定要新增在 `cpp-notes/all problems/`」
* 除非我明確說明：
  -「只要新增在某某資料夾」
  -「不要放進 all problems」
  否則：
* **all problems 一定要新增**
* 其他資料夾（grind75 / neetcode150 / contests 等）只是「額外分類」

### 4.2 多資料夾新增規則

* 若我指定新增文章到某個資料夾（例如 grind75）：

  * 仍需同時在 `all problems` 中新增對應文章
* all problems 為主索引來源
* 其他資料夾視為「主題整理 / 子分類」

### 4.3 重複題號處理規則（重要）

* 若在新增文章時：

  * `cpp-notes/all problems/` 中 **已存在相同題號的文章**
* 則：

  * **不需要再次在 all problems 中新增**
  * 可只新增到指定的其他資料夾（若有說明）

> all problems 中，同一題號最多只會有一篇文章。

### 4.4（新增法律）既有文章補充到其他 C++ 清單資料夾（同文等效規則）

> 目的：你可能一開始只把文章收錄在 all problems，之後才發現它也屬於某個題單（grind75 / leetcode75 / neetcode150 / top-interview-150...）。
> 這種情況下，你會要求「把既有文章新增到某題單資料夾」，而該新增行為在內容上必須等效且一致。

* 當我說「新增 XX 文章到 grind75（舉例）」且該題在 `cpp-notes/all problems/` 已存在時：

  * 這個指令的語意等效於：
    **在指定題單資料夾新增一份對應文章頁（題單版本）**
* 題單版本文章必須遵守：

  * **文章內文必須與 all problems 版本完全一致**
  * 完全一致的範圍包含但不限於：

    * HackMD 原稿的標題文字、段落順序、換行、空白行
    * `# / ## / ### / ####` 層級
    * `---`
    * `**粗體**`、`` `inline code` ``、引用、清單
    * 程式碼區塊內容（不得改寫變數名、不得加註解、不得修正 code）
    * 文末題目連結（依該版本規則，若非 Contest 題解則通常一致）
* 題單版本與 all problems 版本「允許不同」的只有：

  1. 檔案路徑（folder / URL 不同）
  2. breadcrumb / 站內導覽（因路徑不同而不同）
  3. 文章資訊區塊中的「原文連結」（permalink 必須指向各自頁面的永久網址）
* 不允許的差異（明確禁止）：

  * 因為換資料夾就改寫內容、改標題層級、增刪段落、重排原稿
* 完成補充新增後，仍必須依第 6 節更新 `site-index.json` 並完成驗收。

### 4.5（新增法律）LeetCode 題目連結格式必須依資料夾語境分流

#### 4.5.1 all problems 規則（必遵守）

* 放在 `cpp-notes/all problems/` 的 C++ 題解文章：
  `## 🔗 題目連結` 一律使用：

```text
https://leetcode.com/problems/<slug>/
````

* 明確禁止：

  * `/description/`
  * `/contest/weekly-contest-xxx/problems/...`
  * `/contest/biweekly-contest-xxx/problems/...`

#### 4.5.2 contests 規則（必遵守）

* 放在 `cpp-notes/contests/weekly-contest-xxx/` 或 `cpp-notes/contests/biweekly-contest-xxx/` 的題解文章：
  `## 🔗 題目連結` 必須使用對應 contest URL：

```text
https://leetcode.com/contest/weekly-contest-xxx/problems/<slug>/
```

或

```text
https://leetcode.com/contest/biweekly-contest-xxx/problems/<slug>/
```

#### 4.5.3 同題雙版本規則

* 同一題若同時存在：

  * `all problems` 版本
  * `contests` 版本
* 則兩份文章的題目連結**可以不同**，
  但必須各自符合所在資料夾規則。
* 也就是：

  * `all problems` → `/problems/<slug>/`
  * `contests` → `/contest/.../problems/<slug>/`

### 4.6（新增法律）同場 Contest 題解日期一致性

* 同一場 contest 產生的多篇題解文章，若無另外指定，日期應一致。
* 若同一場 contest 的題目同時有：

  * `all problems` 版本
  * `contests` 版本
* 且屬於同次整理，日期也應一致。
* 僅在我明確指定某篇不同日期時，才可例外。

### 4.7（新增法律）題單補收錄時，不能只複製文章檔案

* 當你把既有文章補進其他題單資料夾（例如 `grind75` / `leetcode75` / `neetcode150` / `top-interview-150`）時：
  **不得只複製 html 檔案後就視為完成。**
* 法律上的完成條件至少包含：

  1. 題單資料夾內有對應文章頁
  2. 該題單的 `index.html` 有列出該文章
  3. `site-index.json` 的 `posts` 中有該文章條目
  4. `site-index.json` 的 `folderCounts` 已同步更新
* 少任何一項，均視為未完成。

### 4.8（新增法律）題單補收錄後，分類語意必須完全改乾淨

* 從 `all problems` 複製文章到其他題單後，必須逐項檢查並修正：

  * breadcrumb
  * permalink / canonical / 原文連結
  * 題單名稱
  * 頁首第一行註解
  * 任何殘留的舊資料夾語意
* 明確禁止出現以下情況：

  * 檔案位於 `neetcode150/`，但 breadcrumb 還指向 `all problems`
  * 檔案位於 `grind75/`，但頁首註解仍寫「全部題單文章」
  * permalink / canonical 還是舊資料夾路徑
  * 頁面中殘留 `all problems` / `全部題單文章` / 其他舊分類名稱
* 規則本質：
  **複製文章到題單資料夾後，除了正文可與 all problems 一致，所有分類辨識資訊都必須改成新題單的語意。**

### 4.9（新增法律）題單文章頁第一行註解必須一致且符合資料夾語意

* 同一題單資料夾下的文章頁，第一行註解必須統一格式。
* 例如 `neetcode150` 題單文章頁，註解應採統一格式：

```html
<!-- neetcode150/146-lru-cache.html: NeetCode 150 文章 -->
```

* 不得在同一資料夾內混用：

  * `全部題單文章`
  * `C++ 解題文章`
  * 只寫題名
  * 沒有註解
  * 舊資料夾註解
* 若同資料夾內第一行註解不一致，視為模板與分類語意未清理完成。

### 4.10（新增法律）題單首頁排序必須遵守該題單原始順序

* `all problems` 的排序規則見第 5 節。
* 但 `grind75` / `neetcode150` / `leetcode75` / `top-interview-150` 等題單資料夾：
  **不得直接用題號大小排序，也不得用新增時間排序。**
* 必須遵守：

  * 該題單的原始清單順序
  * 若目前只收錄其中部分題目，也要按照原始清單順序排列「目前已收錄的那些」
* 明確禁止：

  * 新補題目直接加在最後
  * 因為比較方便就改成依數字大小排
  * 同一題單首頁的排序邏輯前後不一致

### 4.11 驗收條件（必做）

1. 新增 C++ 題解時：all problems 必定收錄（除非你明確免除）
2. 題單補收錄時：題單版本與 all problems 版本內文完全一致（僅允許 4.4 所列差異）
3. 若 all problems 已存在同題號：不得重複新增 all problems
4. `all problems` 題目連結符合 `/problems/<slug>/`
5. `contests` 題目連結符合 `/contest/.../problems/<slug>/`
6. 同場 contest 多篇日期一致（除非我明確指定例外）
7. 題單補收錄時，必須同時完成文章頁 / 題單 index / `site-index.json posts` / `folderCounts`
8. 題單補收錄後，breadcrumb / permalink / 註解 / 題單名稱不得殘留舊分類語意
9. 題單首頁排序符合該題單原始順序

## 5) 排序規則（C++ 系統）

### 5.1 all problems 資料夾

* 文章排序依「題號數字」由小到大

  * 例：1, 2, 15, 88, 392, ...
  * `cpp-notes/all problems/index.html` 的顯示順序必須採用「數字比較」排序，不可使用字串（字典序）排序，避免出現 2130 排在 231 前面的情況。

### 5.2 非 all problems 的文章

* 預設排序在 all problems 文章之後
* 若需要特殊排序（例如時間優先、置頂），需另外說明並修改程式邏輯

### 5.3（新增法律）題單首頁排序規則補充

* 若該資料夾屬於明確題單（如 `grind75` / `neetcode150` / `leetcode75` / `top-interview-150`）：

  * 一律依該題單原始清單順序排列
  * 只顯示目前已有文章的那些題目
  * 不得套用 `all problems` 的數字排序規則
* 也就是：

  * `all problems` → 題號數字遞增
  * 題單首頁 → 題單原始順序

### 5.4 驗收條件（必做）

1. all problems 的列表排序符合題號數字遞增
2. 非 all problems 的列表排序規則符合現行程式邏輯或你明確指定的特殊排序
3. 題單首頁的順序符合對應題單原始清單
4. 新補題目不得被直接加在最後

## 6) 類別與計數索引（自動化規則）

### 6.1 索引來源（唯一真相）

* 全站類別與文章計數的資料來源為：`/site-index.json`
* 前端頁面（categories.html / category.html / cpp.html）只讀取 `site-index.json` 來計算，不再於 runtime 呼叫 GitHub API 掃描 repo。

### 6.2 自動更新時機（改版更新一次）

* `site-index.json` 由 GitHub Actions 在每次 push 到 main 後自動重新產生並提交更新。
* 因此：新增/修改文章後，只要成功推到 main 並完成 Actions，類別與數量就會自動更新。

### 6.3（新增法律）本機驗收更新流程（新增文章必做）

* 為了避免 `site-index.json` 落後造成計數錯誤：
  **每次新增/修改文章後，必須先在本機執行** `python3 scripts/build_site_index.py` **更新** `site-index.json`，再 commit/push。
* 即使 GitHub Actions 會在 push main 後重建索引，本機仍必須先跑一次以便：

  * 及早發現文章是否缺 `.post-tag`（會被排除）
  * 及早發現 scope/filter 導致計數異常
  * 讓提交到 repo 的 `site-index.json` 立即可驗收

### 6.4 C++ 學習筆記的特殊範圍規則

* 全站類別計算：包含所有文章頁面（如 home/xxx.html 也算）。
* 但「C++ 學習筆記」相關的類別/文章計數（scope=cpp-notes）僅計算 `cpp-notes/all problems/` 內的文章。

### 6.5（新增法律）計數器不得寫死（你最新要求）

* 除非本法律明確例外，所有頁面上的文章數量/計數器：

  * **必須由 `site-index.json` 計算取得**
  * **禁止硬編碼固定數字**
* scope=cpp-notes 時，僅計入 `cpp-notes/all problems/`（沿用第 6.4）

### 6.6（例外規則）contests 題單頁計數器可寫死

* `cpp-notes/contests/index.html`（Contests 題單頁）中的「每場 contest 題數」：

  * 允許直接寫死數字（因為比完賽題數固定、且該頁通常不動態增補）
* 但此例外**不擴張適用**到其他頁面：

  * `cpp.html`、`categories.html`、`category.html`、以及任何非 contests 題單頁的計數器：仍一律不得寫死（見第 6.5）

### 6.7（補充）前端不得再依賴 repo 掃描做計數

* 若頁面仍存在任何「repo 掃描/呼叫 GitHub API」相關屬性/邏輯：

  * 其不得作為計數來源
  * 計數唯一真相仍是 `site-index.json`

### 6.8（新增法律）題單資料夾必須符合「三一致」

* 對任何有獨立 index 的題單資料夾（如 `all problems` / `grind75` / `neetcode150` / `leetcode75` / `top-interview-150`）：
  都必須滿足以下三者一致：

  1. **資料夾中的實體文章檔案數**
  2. **該資料夾 `index.html` 中的文章卡片數**
  3. **`site-index.json` 中該 folder 的 `folderCounts` 數值**
* 三者只要有一個不一致，視為未完成修改。

### 6.9（新增法律）`folderCounts` 正確還不夠，`posts` 也必須真的有條目

* 不得只把 `folderCounts` 改成看起來正確的數字。
* 還必須檢查：

  * `site-index.json` 的 `posts` 中，真的存在該文章條目
  * 路徑、標題、資料夾資訊皆正確
* 也就是：
  **count 正確 ≠ 文章真的有被索引**
* 法律上必須同時成立：

  * `folderCounts` 正確
  * `posts` 條目存在且正確

### 6.10 驗收條件（必做）

1. 本機執行 `python3 scripts/build_site_index.py` 後，`site-index.json` 立即能反映新增/修改
2. push 到 main 後 Actions 成功，且 `site-index.json` 確實更新並提交到 repo
3. 除 contests 題單頁外，任何計數器不得出現硬編碼數字
4. C++ scope 計數僅使用 all problems 文章數量
5. 每個有改動的題單資料夾都符合「實體檔案數 / index 卡片數 / folderCounts」三一致
6. 相關文章不只 count 對，`posts` 也必須真的存在條目

### 7.1 資源統一位置（唯一規範）

* 全站共用圖片與靜態資源一律放在：`/assets/`

### 7.2 路徑引用規則（必遵守）

  * ❌ `assets/hero-1.jpg`

### 7.3 Hero 圖片命名規則（首頁全螢幕大圖）

* Hero 輪播圖片固定命名：

  * `/assets/hero-1.jpg`
  * `/assets/hero-2.jpg`
  * `/assets/hero-3.jpg`
* 若未來新增更多張，沿用：

  * `/assets/hero-4.jpg`、`/assets/hero-5.jpg` ……

> 規則：**檔名與路徑優先固定**；更換圖片內容可以，但不要改命名規則，避免每次都要改前端程式。

### 7.4 Avatar 命名規則（個人檔案頭像）

* 個人頭像固定命名：

  * `/assets/avatar.jpg`
* 允許換照片內容，但 **檔名與路徑不得改**

### 7.5 改名/搬移資源後的必做檢查（驗收條件）

任何修改 assets（新增/刪除/改名/搬位置）後，必須做到：

1. 全 repo 搜尋舊檔名（例如 `hero-` / `avatar`）確保沒有殘留引用
2. 確認所有引用都是 `/assets/...`
3. 部署後以 Ctrl+F5 強制刷新確認圖片正常顯示

## 8) 文章頁面規格（以 Two Sum 為唯一標準）

> 目的：確保任何新文章看起來、結構與資訊都一致，並且一定能被類別系統索引。

### 8.1 文章頁面定義

* 任何希望被 `site-index.json` 納入的頁面，都必須符合本節規格
* 不符合者視為「非文章頁」，不得納入類別系統與分類統計

### 8.2 必備結構（必遵守）

每一篇文章必須具備（以 Two Sum 為標準）：

1. **文章內容區**（解題說明/心得/整理）
2. **目錄 TOC**（文章長度足夠時要有；格式與 Two Sum 一致）
3. **文章資訊區塊（文章資訊）**：必須放在文章下方，長得要像 Two Sum 的「文章資訊」卡片

> 規則：三者缺一不可（對應第 3.3）。

### 8.3 「文章資訊」區塊必填欄位（跟 Two Sum 一樣）

文章底部的「文章資訊」必須包含：

* 作者（例如 scottnick）
* 撰寫日期（或更新日期，至少要有一個日期欄位）
* 類別/標籤（以 badge 顯示，且每個標籤同時是 `.post-tag`）
* 原文連結（本篇文章的永久網址）

> 規則：新增文章時 **一定要補齊文章資訊**，否則視為不完整文章。

### 8.4 `.post-tag` 規則（類別索引唯一標準）

* 每篇文章必須至少有 1 個 `.post-tag`
* `.post-tag` 文字必須乾淨（避免多餘空白/奇怪符號）
* 可多標籤（代表多分類）
* 沒有 `.post-tag` 的頁面不納入類別系統

### 8.5（新增法律）頁首第一行註解也是文章頁模板的一部分

* 正式文章頁的第一行註解，視為模板結構的一部分。
* 同一資料夾內的文章頁，第一行註解格式應一致。
* 不得出現：

  * 有的頁面有註解、有的沒有
  * 有的寫資料夾語意、有的只寫題名
  * 有的仍殘留舊資料夾註解
* 若同資料夾內第一行註解不一致，視為該資料夾模板未統一完成。

### 8.6 新增文章後的驗收條件（Checklist）

新增文章後必須確認：

1. 文章頁可正常開啟（無 404）
2. 文章底部「文章資訊」完整（作者/日期/標籤/原文連結）
3. `.post-tag` 存在且正確
4. 【法律】新增文章後必須觸發第 6 節索引更新流程

   * 必須先依第 6.3 在本機更新 `site-index.json`
   * 必須 push 到 `main`
   * 必須確保 GitHub Actions 有成功執行
   * 必須確保 `site-index.json` 已被 Actions 重新產生並更新（提交到 repo）
   * 若 Actions 失敗或 `site-index.json` 未更新：本次「新增文章」視為未完成，必須修到成功為止
5. 類別頁與計數能找到此文章（以 `site-index.json` 為唯一真相）
6. Ctrl+F5 強制刷新後仍顯示正常（避免快取造成誤判）
7. 頁首第一行註解符合該資料夾語意，且與同資料夾其他文章一致

### 8.7（新增法律）單篇文章頁固定區塊責任分工（必遵守）

* 單篇文章頁中的主要固定區塊，必須有明確且穩定的責任分工，不得混用。
* 各區塊責任如下：

  * `hero`：負責頁首標題資訊
  * `.article-toc` / TOC 容器：負責目錄導覽
  * `.article-content`：負責文章正文
  * `article-footer` / 文章資訊區塊：負責文章底部固定資訊
* 規則本質：

  * 頁首資訊屬於頁首
  * 正文屬於正文
  * 文章資訊卡屬於底部固定資訊
  * 三者不得互相取代，也不得重複承載同一份內容

### 8.8（新增法律）單篇文章頁唯一性規則（必遵守）

* 單篇文章頁中，下列主要結構都應視為**唯一區塊**，不得重複出現：

  * `hero`
  * `.article-toc` 主容器
  * `.article-content`
  * `article-footer`
* 單篇文章頁只能有 **一個主要 `h1`**。
* 這個主要 `h1` 必須位於頁首主區塊（通常是 `hero`）中。
* 不得於 `.article-content` 再出現第二個文章主標題 `h1`。
* 若頁首已顯示一份標題 / 日期 / 難度 / 類型等 metadata，正文區不得再次重複渲染同一組資訊。

### 8.9（新增法律）`hero` 與 `.article-content` 的嚴格分界（必遵守）

* `hero` 屬於頁首摘要區塊，不屬於文章正文。
* `.article-content` 屬於文章正文，不得被拿來重做第二組頁首摘要區塊。
* 明確禁止以下情況：

  * 頁面已有 `hero`，正文開頭又再放一個 `h1`
  * 頁面已有 `hero` metadata，正文開頭又再放一次練習日期 / 難度 / 類型
  * 頁面已有 `hero`，正文最前面仍保留第二組標題摘要卡
* 若出現上述情況，視為模板結構錯誤，而非單純排版問題。

### 8.10（新增法律）LeetCode / 題解頁正文起點規則（必遵守）

* 若該頁屬於 LeetCode / 演算法題解頁，且頁首資訊已由網站模板承接到 `hero`：

  * `.article-content` 應直接從正文開始
  * 原則上應從第一個 `##` 段落開始（例如 `## 📘 題目敘述`）
* 不得為了「看起來完整」而在正文區補回第二組頁首資訊。
* 此規則只是在網站 HTML 落位上規範固定區塊與正文的分工，
  **不代表可以修改 HackMD 原稿本身**。

### 8.11（新增法律）模板一致性不只看畫面，也看 DOM 骨架

* 單篇文章頁模板一致性，不只看畫面是否相似，也看 HTML / DOM 骨架是否一致。
* 同類型頁面至少應維持以下順序與語意穩定：

  1. site header
  2. breadcrumb
  3. `hero`
  4. TOC 容器
  5. `.article-content`
  6. `article-footer`
  7. `.post-tag`（若獨立存在於資訊卡附近）
* 若某頁畫面看起來像文章頁，但底層結構把頁首資訊、正文、底部固定資訊混在一起，仍視為不符合模板。

### 8.12（新增法律）模板類修改後的跨頁抽查（必做）

* 若本次修改涉及以下任一項：

  * 單篇文章頁模板
  * `hero`
  * TOC 容器
  * `.article-content`
  * `article-footer`
  * 頁首 metadata 顯示方式
  * HackMD → HTML 的落位方式
* 則不得只看單一頁面。
* 至少必須抽查：

  1. 一篇 `all problems`
  2. 一篇 `contests`
  3. 一篇其他正常單篇文章頁
* 驗收時必須確認：

  * 主 `h1` 只有一個
  * `hero` 與 `.article-content` 沒有重複承載同一份頁首資訊
  * `.article-content` 沒有被塞入第二組 header / hero 型區塊
  * `article-footer` 與 `.post-tag` 未被破壞

## 9) 文章頁面內容輸入格式（HackMD 原稿 → 網站文章頁）【新增法律】

> 本節規範兩件事：
>
> 1. **HackMD 原稿本身怎麼寫**
> 2. **HackMD 原稿轉成網站文章頁時怎麼呈現**
>
> 最核心原則：**HackMD 原稿是文章本體，網站只負責正確呈現，不得任意改寫原稿內容。**

### 9.0 最高優先級規則（必遵守）

* **我上船的格式請完全遵守，不要自己刪除文字或修正。**
* 「上船格式」包含但不限於：

  * 標題文字、段落順序、換行、空白行
  * `# / ## / ### / ####`
  * `---`
  * `**粗體**`
  * `` `inline code` ``
  * `>` 引用
  * `- / *` 清單
  * 程式碼區塊 ` `
  * emoji（📘🧠🪜等）
  * 文末連結原文（直接貼 URL 也要保留）
* 允許做的事情只有：

  * **把 HackMD 語法「呈現在網頁上」**（轉成對應的 HTML / 樣式）
  * **補上網站必要的固定區塊**（例如文章資訊區塊、站內導覽、TOC 容器）
* 不允許做的事情：

  * 改寫句子、補字、刪字、修正文案、重排內容
  * 把原本寫的段落合併 / 拆開
  * 把原本寫的標題層級改掉（例如把 `###` 改成 `##`）
  * 因為比較順手、比較美觀、比較像官方題解，就改寫成另一種演算法、另一種文章骨架、另一種說法
  * 因為要配合網站模板、分類、SEO、資料夾結構，就反向要求修改原稿內容

### 9.0.1 標題文字與符號必須完全照原文

* 新增文章頁面時，文章內容區的標題（包含但不限於 `# / ## / ### / ####` 的標題文字）**必須完全照我提供的原文輸出**。
* 禁止任何自動修正或改寫，包括但不限於：

  * 增加 / 刪除標點符號（例如冒號、括號、破折號、句點）
  * 更改大小寫、空白數量、emoji
  * 替換同義字或改成更「順」的標題
* 允許做的只有：

  * 依網站呈現需要補 `id`（供 TOC 跳轉），但不得改動標題顯示文字本體。

### 9.0.2 原稿優先於網站轉換，不得反向改稿

* HackMD 原稿屬於文章本體，網站頁面的責任是「忠實呈現」。
* 不得因為：

  * 網站模板
  * TOC
  * 排版需求
  * 分類頁需求
  * SEO / canonical / metadata 需求
  * 資料夾位置差異
  * all problems / contests / 題單版本差異
    就反向修改原稿內容。
* 即使某篇舊文章的骨架與新規格不同：

  * **網站轉換時仍應以原稿為準**
  * 不得為了硬套新骨架而增刪段落、改標題層級、改內容順序
* 規則本質：

  * **新規格主要約束新撰寫的原稿**
  * **舊原稿仍受「不得改稿」保護**

### 9.1 文章原始稿來源

* 我新增 / 撰寫文章內容時，預設會用 **HackMD 的 Markdown 語法**輸入（我稱為「Hack 格式」）。
* 網站文章頁必須支援並正確轉換 / 呈現 HackMD 語法（至少支援本節列出的規則）。

### 9.1.1 HackMD 原稿分成兩種類型

* HackMD 原稿可分為以下兩類：

  1. **一般文章原稿**
  2. **LeetCode / 演算法題解原稿**

### 9.2 標題層級與目錄（TOC）規則（必遵守）

### 9.2.1（新增法律）TOC 的唯一真相：文章內的 `## (h2)`

* **目錄（TOC）一定只收錄文章內容區 `.article-content` 裡的 `h2`（也就是 Markdown 的 `##`）。**

### 9.2.3（新增法律）TOC 解析範圍與 ID 規則

### 9.2.4 驗收條件（必做）

## 9.11 LeetCode / 演算法題解原稿的寫作目的與總原則【新增法律】

> 最重要：題解筆記的目的，是讓我自己之後回來看時，能快速看懂自己的程式碼在做什麼；
> 同時也讓別人只看文章，就能理解這份 code 的主要邏輯。
>
> 不追求炫技，不追求寫成官方題解風格；
> **重點是：好理解、能對上 code、和原始程式碼高度一致。**

### 9.11.1 全文視角與語氣規則

* LeetCode / 演算法題解原稿，全文預設使用 **第一人稱**。
* 應使用這類表述：

  * 我這份寫法……
  * 我的想法是……
  * 我先……
  * 我用……
* 不得使用這類表述：

  * 你的想法……
  * 你這份程式……
  * 你這裡……
* 語氣應：

  * 白話、自然、便於理解
  * 像在整理自己的解法筆記
* 不得：

  * 寫成在批改使用者
  * 寫成過度隨便的聊天口吻
  * 寫成抽離 code 的泛泛題解

### 9.11.2 內容必須貼著原始 code 寫

* 題解文章的內容必須與原始程式碼 **高度對應**。
* 也就是：

  * code 用 stack，就寫 stack
  * code 用 prefix sum，就寫 prefix sum
  * code 有排序，就要講排序
  * code 有特判，就要照實寫出來
  * code 用 DP，就要依照實際狀態寫 DP
* 不得：

  * 偷換成別種演算法
  * 補不存在的函式、資料結構、優化版本
  * 擅自改變變數名、流程順序、狀態定義
* 規則本質：
  **文章是在幫我整理這份 code，不是在重寫一份理想解。**

### 9.11.3 不要自己補我沒有寫的東西

* 不得自己補：

  * 我沒有寫的變數
  * 我沒有寫的函式
  * 我沒有做的優化版本
  * 我沒有使用的演算法
  * 我沒有使用的資料結構
* 若原始 code 是暴力解，就照暴力解寫。
* 若原始 code 是某種特定寫法，就照那種寫法寫。
* 不得因為覺得更漂亮、比較標準、比較常見，就把內容改寫成另一個版本。

### 9.11.4 題外平台要求不得寫入筆記

* 題目中若包含平台額外要求，例如：

  * `Create the variable named ...`
  * `請在函式中建立某個特定名稱的變數`
* 這類內容 **一律不得寫進筆記原稿**。
* 原因：

  * 這不是題目本質
  * 會污染筆記可讀性
  * 和理解 code 沒有直接幫助

### 9.11.5 不得加入與 code 無關的多餘元敘述

* 以下這類句子原則上不得隨意加入，除非真的對理解有幫助：

  * 這題很簡單
  * 這題很經典
  * 這題很妙
  * 這是標準做法
* 也不得加入無意義括號說明，例如：

  * （leetcode原文翻譯）
  * （這裡是 h3）
  * （這邊就是某某意思）

## 14)（新增法律）單篇文章頁 HTML 原始碼排版與可維護性規則

> 目的：避免畫面雖然正常，但 HTML 原始碼被壓成一坨，導致後續維護、比對 diff、人工檢查都困難。
> 本條規則要求的是「基本可讀性與可維護性」，不是要求所有 HTML 都統一成更嚴格的 formatter 風格。

### 14.1 HTML 修改後必須保持基本可讀排版

* 任何 HTML 檔修改後，原始碼必須保持基本可讀性。
* 不得故意將大量結構壓成單行或難以維護的一坨。
* 但也**不要求**為了格式美化而額外重排整份 HTML。

### 14.2 可接受的排版標準

* HTML 原始碼只需達到「既有正常文章」的可讀程度即可，即視為合格。

* 本站目前可接受的參考樣板為：

  * `cpp-notes/grind75/1-two-sum.html`

* 也就是說，只要原始碼的可讀性與結構清楚程度達到這篇的水準，就不需要再額外調整。

### 14.3 必須避免的情況

以下情況視為不合格：

* `<head>`、`<header>`、`<main>`、`<section>`、`article-footer` 等主要結構大段擠成一行
* 大量區塊沒有基本縮排，導致難以人工閱讀
* 為了快速修改而把整段 HTML 壓縮成難以維護的格式

### 14.4 不得為了格式整理而改動內容

* 不得為了整理 HTML 排版而改動以下內容：

  * 文章文字
  * 題目連結
  * 日期
  * metadata
  * `.post-tag`
  * DOM 結構語意
  * 程式碼內容

* 規則重點是「整理可讀性」，不是重寫頁面。

### 14.5 `<pre><code>` 區塊不得被排版工具破壞

* 可以整理 `<pre><code>` 的外層縮排
* 但不得修改其內部程式碼內容，包括：

  * 原本縮排
  * 原本換行
  * 原本空白
  * 原本字元內容

### 14.6 HTML 排版也屬於驗收項目

* 即使畫面功能正確，若原始碼可讀性明顯過差，仍視為未完成修改。
* 但若原始碼已達到本網站既有正常文章（如 `cpp-notes/grind75/1-two-sum.html`）的可讀程度，則不得再要求額外 formatter 式美化。

### 14.7 驗收條件（必做）

1. 修改後的 HTML 不再出現大段主要結構全擠成一行
2. 結構區塊有基本可讀的縮排與分行
3. `<pre><code>` 內容未被破壞
4. 原始碼可讀程度至少達到 `cpp-notes/grind75/1-two-sum.html` 的水準
5. 不得為了格式整理而額外改動文章內容或結構

### 14.8（新增法律）HTML 原始碼一致性不只看排版，也看結構語意

* HTML 原始碼的「統一」不只指縮排與換行，還包括：

  * 區塊順序一致
  * 區塊語意一致
  * 固定區塊責任一致
  * DOM 骨架一致
* 同類型單篇文章頁，應儘量維持相同骨架。
* 不得出現：

  * 某些頁面把 `hero` 放在正文裡
  * 某些頁面把正文開頭做成第二組 header
  * 某些頁面缺 `article-footer`，卻把其內容散落在正文裡
* 規則本質：

  * HTML 原始碼是否正確，不只看畫面能不能顯示
  * 也看後續維護時，能否清楚辨識每個區塊的責任

### 14.9（新增法律）單篇文章頁原始碼骨架應穩定

* 單篇文章頁原始碼骨架應維持穩定順序，原則上應為：

  1. `head`
  2. site header
  3. breadcrumb
  4. `hero`
  5. TOC 容器
  6. `.article-content`
  7. `article-footer`
  8. `.post-tag`（若獨立）
  9. script / 頁尾
* 若既有結構有微幅差異，可不強制逐字一致，
  但必須維持：

  * `hero` 在正文前
  * TOC 與正文的關係穩定
  * 文章資訊卡在正文後
  * 不得產生第二組 header 型區塊混進正文

### 14.10（新增法律）若模板錯誤會造成重複標題，視為結構性錯誤

* 單篇文章頁若因模板輸出錯誤，造成：

  * 兩個主 `h1`
  * 兩組頁首 metadata
  * `hero` 與正文重複顯示同內容
* 則視為**結構性錯誤**，不是單純排版問題。
* 這類錯誤修正後，必須依第 0.7 同步升級法律或驗收流程。

## 15)（新增法律）單篇文章頁 header / 類別按鈕模板規則

> 目的：把你這次反覆修正的 banner / 類別按鈕模板正式寫成網站法律，避免之後再次混用多套樣式。

### 15.1 單篇文章頁的上方模板必須一致

* 所有單篇文章頁面的上方區塊，包含：

  * site header
  * breadcrumb
  * hero 區塊
  * 右上角「類別」按鈕
* 必須使用統一模板。

### 15.2 `.nav-action-link` 類別按鈕不得混用多套版本

* 以下都視為舊格式或錯誤格式，不得再出現：

  * 純文字 `類別`
  * `<span>類別</span>` 混用版
  * 不同座標/不同樣式的另一套 SVG icon
* `all problems` 與 `contests` 的差異只允許：

  * `href="../../categories.html"` vs `href="../../../categories.html"`
* 除路徑層級外，HTML 結構、icon、文字格式都必須一致。

### 15.3 標準模板必須作為唯一真相

* 只要同屬於「單篇文章頁」，就必須比照主流正確模板。
* 不得再以其他舊文章當作例外模板。

### 15.4 驗收條件（必做）

1. 不再出現純文字 `類別`
2. 不再出現多套 SVG icon 混用
3. `all problems` 與 `contests` 只差路徑層級
4. 單篇文章頁上方區塊模板完全一致

### 15.5（新增法律）`hero` 與 breadcrumb 也屬於單篇文章頁模板的一部分

* 單篇文章頁模板不只包含 `.nav-action-link`，也包含：

  * breadcrumb
  * `hero`
  * `hero` 中的主標題與頁首摘要資訊
* 因此，若修改：

  * breadcrumb HTML
  * `hero` 結構
  * `hero` 中 metadata 顯示方式
* 也必須視為模板類修改，適用第 0.6、8.12、14.10 的跨檔案驗收規則。

### 15.6（新增法律）單篇文章頁的主 `h1` 必須統一由頁首模板承載

* 單篇文章頁的主 `h1` 必須統一由頁首模板承載。
* 不得有的頁面把主 `h1` 放在 `hero`，有的頁面把主 `h1` 放在 `.article-content`。
* 若某些舊頁仍存在不同作法，後續修正時應往主流正確模板收斂，而不是繼續擴散多套作法。

### 15.7（新增法律）頁首模板與正文不得互相侵入

* header / breadcrumb / `hero` 屬於頁首模板。
* `.article-content` 屬於正文。
* 兩者必須維持語意邊界，不得互相侵入。
* 例如：

  * 不得把第二組 `h1` 塞進正文
  * 不得把正文第一段包進 `hero`
  * 不得把 breadcrumb 或 metadata 當成正文段落輸出

### 15.8（新增法律）標準模板的驗收要同時看畫面與原始碼

* 頁首模板是否一致，不得只靠肉眼看畫面。
* 驗收時應同時確認：

  * HTML 結構
  * class 命名
  * DOM 骨架
  * `hero / breadcrumb / nav-action-link` 的相對順序
* 若畫面相似但 HTML 明顯不同，仍視為模板不一致。

## 16)（新增法律）題單頁、SITE INDEX 與收錄完整性規則

> 目的：把這次實際修到的 `grind75` / `neetcode150` / `site-index.json` / 缺文章 / 排序 / 計數器問題正式寫成法律，避免未來再次發生同類錯誤。

### 16.1 題單「已收錄」的法律定義

* 當你說某題「已收錄到某題單」時，法律上的意思必須是：

  1. 題單資料夾內有對應文章頁
  2. 題單首頁 `index.html` 有該文章卡片
  3. `site-index.json > posts` 有該文章條目
  4. `site-index.json > folderCounts` 已正確更新
* 只有其中一項成立，不得稱為「已收錄」。

### 16.2 題單首頁漏列文章視為嚴重錯誤

* 若題單資料夾內已有文章頁，但題單首頁 `index.html` 沒有列出：

  * 視為未完成修改
  * 不得以「反正檔案存在」當作完成理由
* 反過來，若 `index.html` 有卡片但實體文章不存在：

  * 視為壞連結風險
  * 同樣視為未完成修改

### 16.3 `site-index.json` 不得只改數字假裝同步

* 不得只更新：

  * `folderCounts`
  * 或頁面上顯示的數字
* 而未同步更新：

  * `posts`
  * 題單首頁卡片
  * 實體文章
* 任何「只改表面數字」的做法，都視為違法修改。

### 16.4 題單首頁與 SITE INDEX 的驗收，必須以資料夾為單位檢查

* 只要本次修改有動到題單資料夾（例如 `grind75` / `neetcode150`）：
  就必須以該資料夾為單位，同時檢查：

  * 實體文章數
  * `index.html` 卡片數
  * `site-index.json` 的 `folderCounts`
  * `site-index.json` 的 `posts` 條目存在性
* 不得只抽查單篇就視為該資料夾驗收完成。

### 16.5 題單補收錄新增文章時，來源文章內文不得縮水

* 若題單文章頁是從既有文章（通常是 `all problems`）補收錄而來：

  * 內文必須等同來源文章
  * 不得因複製到題單就被刪段、少標題、少 code block、少分隔線
* 若工具（例如自動化工具 / codex 類工具）有刪改內容風險，必須特別人工比對。

### 16.6 題單首頁的正確排序，屬於法律要求，不是美觀問題

* 題單首頁排序錯誤，視為結構性錯誤，不得視為單純 UI 問題。
* 原因：

  * 題單本身就是依某份清單邏輯整理
  * 排序錯誤會直接破壞題單語意
* 因此，排序錯誤必須視為法律違規，而不是「之後再調美觀」。

### 16.7 題單頁第一行註解不一致，視為分類模板未清理完成

* 若題單資料夾內：

  * 有的頁面第一行註解寫 `NeetCode 150 文章`
  * 有的寫 `全部題單文章`
  * 有的沒有註解
  * 有的寫 `C++ 解題文章`
* 則視為：

  * 題單模板未統一
  * 題單分類語意未清乾淨
* 必須修正後才算完成。

### 16.8 驗收條件（必做）

1. 題單中每篇新增/補收錄文章都符合 16.1 的「已收錄」定義
2. 不存在資料夾有文章但首頁沒列、或首頁有列但檔案不存在的情況
3. `folderCounts` 與 `posts` 同時正確
4. 題單補收錄文章內文未縮水、未被工具亂刪
5. 題單首頁排序符合原始題單順序
6. 題單內第一行註解一致且符合資料夾語意

### 16.9（新增法律）題單補收錄文章也必須符合單篇文章頁模板完整性

* 題單資料夾中的文章，雖然可能是從 `all problems` 複製 / 補收錄而來，
  但仍必須完整符合單篇文章頁模板規則。
* 不得出現：

  * 題單版本少了 `hero`
  * 題單版本少了文章資訊卡
  * 題單版本缺 `.post-tag`
  * 題單版本正文多出第二組頁首摘要
* 也就是：

  * 題單補收錄不只是收錄問題
  * 也是文章模板一致性問題
